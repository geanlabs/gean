# Don't-touch list

Files and patterns this skill must **not** analyze or suggest changes to.
If a finding falls into any of these, drop it before reporting. If the
entire diff is don't-touch, say so and stop — there's nothing to review.

---

## Hard skip — never analyze

### Generated code

- `internal/types/*_encoding.go` — generated by `make sszgen` from struct tags.
  Will be overwritten on the next regen. Any "simplification" here is
  reverted the moment someone runs `make sszgen`. These files are **only ever
  produced by `make sszgen`** — never hand-written or hand-edited. If you see a
  hand-authored `*_encoding.go` (or any by-hand SSZ codec) in the diff, that is
  a finding (`duplicates-existing-util`): the fix is to delete it, edit the
  struct + tags, and run `make sszgen`. If `make sszgen` fails, report the
  error and stop — do not patch the file by hand.
- Any file with a `// Code generated by ... DO NOT EDIT.` header.
- `xmss/rust/target/` — Rust build output (also gitignored, shouldn't appear).

### Foreign-language code

- `xmss/rust/**/*.rs` — Rust source. Different review philosophy, different
  tooling (cargo fmt, clippy). Out of scope for a Go-focused lean review.
- `xmss/rust/**/Cargo.toml` and `Cargo.lock`.
- Any `*.py` files (leanSpec fixture generators, lean_compiler sources).

### Fixture parsers

- `internal/specfixtures/*.go` — types and field names mirror the JSON shape of
  leanSpec fixtures. "Unused" fields are required for JSON unmarshaling to
  succeed. Renaming or restructuring breaks fixture loading.
- `internal/spectests/fixture.go` — same reasoning; mirrors fixture JSON. Field-by-
  field equivalence with the spec generator is the point.

### Vendored or third-party

- `vendor/` (if present).
- Anything under a path matching `third_party/` or `external/`.

## Soft skip — analyze, but only report, never auto-apply

Findings in these areas can appear in the report (a reviewer may still want
to see them), but they must be tagged `consensus-critical` and never
applied without explicit per-finding human approval — even in fix mode.

### Spec-mirroring function bodies

- `internal/statetransition/transition.go`, `internal/statetransition/slots.go`,
  `internal/statetransition/block.go`, `internal/statetransition/attestations.go`,
  `internal/statetransition/justifiable.go` — function names and structure mirror
  leanSpec. Restructuring (even if it preserves behavior) destroys
  auditability against the spec. Reviewers should be able to put gean's
  `ProcessAttestations` next to leanSpec's `process_attestations` and read
  them in parallel.
- Cross-check with `.claude/skills/spec-compliant/references/spec-gean-mapping.md`
  before suggesting any structural change here. If the function name maps
  to a named spec function, tag `spec-divergence-risk` and refuse to apply.

### Fork choice algorithm

- `internal/forkchoice/protoarray.go` — `ApplyScoreChanges`, `OnBlock`, `Prune`,
  best-child/best-descendant recompute. Mirrors the canonical proto-array
  algorithm; deviation here is high-risk.
- `internal/forkchoice/votes.go` — `RemapIndices` looks ceremonial but is silently
  load-bearing. Never propose deleting "unused" remap logic without proving
  via test that head computation still converges after a prune.

### Consensus engine

- `internal/node/` single-writer loop — `engine.go`, `dispatch.go`, `tick.go`,
  `gossip.go`, `head.go`, `import.go`, `proposal.go`, `duties.go`, `gate.go`,
  `fetch.go`, `workers.go`, `replay.go` — plus `internal/pending/`
  (`blocks.go`, `attestations.go`), `internal/dutygate/` (`gate.go`,
  `policy.go`), `internal/role/controller.go`, and
  `internal/store/consensus_store.go`. The single-writer engine and the
  interval-driven tick loop are consensus-critical. Even a "harmless" inline of
  a helper can change goroutine scheduling under load.

### P2P gossip and req/resp

- `internal/p2p/host.go` — gossipsub configuration. The mesh size, heartbeat,
  duplicate cache, and `StrictNoSign` choices are interop-critical.
  Defaults that look bloated are actually negotiated with other Lean
  clients.
- `internal/p2p/reqresp.go` — Status (80-byte payload), BlocksByRoot, BlocksByRange.
  Wire format must match other clients exactly.
- `internal/p2p/topics.go` — topic strings and fork digest. Changing these breaks
  gossip with every peer.

### Core types

- `internal/types/state.go`, `internal/types/block.go`, `internal/types/attestation.go`,
  `internal/types/checkpoint.go`, `internal/types/validator.go`, `internal/types/config.go`,
  `internal/types/constants.go` — SSZ struct shapes and constants. Any field change
  requires `make sszgen` and a spec-conformance check. Constants like
  `SecondsPerSlot`, `IntervalsPerSlot`, `HistoricalRootsLimit` are
  network-wide; changing them forks the chain.

### XMSS Go binding

- `xmss/ffi.go` — CGo declarations. Changing function signatures or buffer
  ownership rules can corrupt memory.
- `xmss/keys.go` — dual-key routing. The split between attestation key and
  proposal key is a one-time-signature reuse guard, not an arbitrary
  design choice.
- `xmss/proof_pool.go` — buffer pool sizing tuned to leanMultisig proof
  size. Changes need benchmarks.

### Spec-conformance test harness

- `internal/spectests/stf_test.go`, `internal/spectests/forkchoice_test.go`,
  `internal/spectests/signatures_test.go`, and friends. These exist to fail loudly
  when gean diverges from spec. "Cleaning them up" risks weakening the
  signal.

## Always-review

Everything else, including:

- `internal/syncer/` (backfill, driver, peer, status) and `internal/metrics/`
  (non-engine paths — still risky, but analyzable)
- `internal/storage/` — Pebble adapter, memory adapter, key encoding
- `internal/genesis/` — config loading and genesis state construction
- `internal/checkpoint/` — checkpoint sync fetcher and verifier
- `cmd/` — main, keygen, CLI flag parsing
- `internal/api/` — HTTP handlers (admin, finalized state/block, fork choice JSON)
- `logger/` — tagged logger
- New test files (`*_test.go`) outside `internal/spectests/`
- New helpers and utilities anywhere

For these, normal risk classification applies (cosmetic / structural /
consensus-critical based on the actual call graph).

## How to use this file from the skill

Before analyzing a file:

1. Match the file path against the **hard skip** list. If it matches, drop
   the file entirely.
2. If a finding lands in a **soft skip** area, do not drop the finding —
   include it in the report, tag it `consensus-critical`, and add the
   explicit "report-only, requires per-finding human approval" note.
3. Otherwise, classify risk based on the actual file path per SKILL.md.

When in doubt, treat unfamiliar code as soft-skip rather than always-review.
A false "consensus-critical" tag wastes some reviewer attention; a false
"structural" tag on consensus code can cause a fork.
